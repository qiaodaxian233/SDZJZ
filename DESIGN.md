## 核心愿景（作者口述 · 优先级高于细节文档）

一句话：**给不会搭红石/生电的人用的「节点工厂」**——一块画布，一头接入（输入），中间拖几个模块连起来，另一头直接出材料。照着教程都搭不对生电的那批人，连几个节点就能量产。

- 界面 = ComfyUI 式节点画布，**整个交互就是画布（B 方案）**，不是表单套画布。
- **不把「不免费复制 / 无限仓储平衡」当硬约束**（推翻《完整设计文档》§12 的硬性平衡）——默认好上手、能出东西；要材料成本的人留 config 开关。
- 《完整设计文档.md》里的多方块矩阵 / 喷射 / 比较器 / 无限仓储等，降级为**可选扩展层**；核心先把「输入 → 模块 → 输出」的节点管线做爽。

## 一、本轮已拍板的三个方向

1. **架构**：控制器方块 + 可连线节点画布（最像 ComfyUI，也是本项目技术难度最高的一块）。
2. **成本模型**：电力系统（发电面板供电，字面呼应"生电"）。
3. **当前阶段**：先出本设计文档。

---

## 二、核心架构：三大件

### 1) 控制器方块（枢纽 / Terminal）
- 世界里放一个控制器方块，右键开「节点画布」GUI。
- 一个控制器管理一张画布（一组面板节点 + 连线 + 电网缓冲）。**服务端持有真实状态，客户端只渲染**。
- 大型生产做成**多方块**：控制器 + 若干「机箱 / 扩展位」方块，扩展位越多，画布容量 / 面板槽越多——给"超大工作台"一个物理体量与合成门槛。

### 2) 节点画布 GUI（面板即节点）
- 画布上每个**面板 = 一个节点**，有输入口 / 输出口。
- **连线 = 物料流 / 电力流**：A 的输出口连到 B 的输入口，A 的产物喂给 B。
- 全项目难度最高的一块（拖拽、贝塞尔连线、缩放平移、端口命中检测全得自绘）→ 分期做，见「四」。

### 3) 面板（节点）分类
- **发电面板**：耗燃料 / 资源 → 电。
- **生产面板**：耗电（+ 可选输入）→ 产出基础物（矿锭 / 食物 / 刷怪掉落…）。
- **加工面板**：A + 电 → B（熔炼 / 压制 / 合成链）。
- **采集面板**：模拟刷怪塔 / 农场 / 挖矿的产出（掉落表 / 农作物 / 矿物）。
- **传输·存储面板**：缓冲、与相邻容器 / 管道对接、把产物导出到世界。
- **控制·逻辑面板**（后期）：过滤 / 分流 / 限量 / 条件开关，对应 ComfyUI 的逻辑节点。

### 4) 加速模块（升级件）
- 插进面板的升级子槽，缩短间隔 / 提高产量。
- 升级槽有上限（建议 4）。
- **提速 = 超线性拉高耗电**（见电力系统），天然刹车，防无脑堆叠。

---

## 三、电力系统（呼应"生电"）

- 电单位：暂定「焦耳 / J」或直接叫「电」，容量 / 流量全走 config。
- 发电面板产电进入控制器的**电网缓冲**；生产 / 加工面板每次运转扣电。
- **电不足 → 面板暂停**（不报废、不掉产物、不刷屏）。
- 加速模块提速的耗电按**超线性**增长（例：×2 速 → ×2.5~3 耗电），逼你扩发电，形成生产链纵深。
- 价值：主题贴合；防"合成个面板就白嫖无限量产"；给系统一个清晰的资源汇（sink）。

---

## 四、节点画布落地：难点 + 诚实分期

拖拽式节点画布在 Fabric 里要**全自绘**，一步到位风险高，且 Phase 1 就卡在 UI 上不划算。策略：

- **MVP（Phase 1~3）**：先**不做**自由拖拽画布，用**槽位机架式 GUI**——控制器 GUI 里一排面板槽，每个面板带 I/O 与升级子槽；"连线"先用**目标选择**实现（面板配置里选"我的产物送到 X 号面板 / 送到相邻容器"）。功能上等价于连线，先把电力 / 生产 / 加速 / 缓冲跑通。
- **进阶（Phase 6+）**：机架升级成真正的**节点画布**——自定义 widget 画节点卡片 + 贝塞尔连线 + 拖拽 / 缩放 / 平移 + 端口命中检测。

> **关键**：数据模型从第一天起就按「节点 + 边(edge)」存，这样后期只换**渲染层**、不动**逻辑层**。

---

## 五、防卡顿（一等公民）

- 产物**只进内部缓冲，缓冲满即暂停，绝不生成掉落物实体**（掉落物刷屏是头号卡顿源）。
- 单面板 + 全局吞吐上限，config 可调。
- 面板按间隔触发、错峰调度，**不每 tick 全量遍历**。
- 服务端算、客户端显示，电量 / 进度走网络包同步。
- 大画布 / 多控制器场景做节流与合并 tick。

---

## 六、数据与持久化

- 面板配置（配方 id、升级数量、缓冲、连线关系）用 **数据组件（Data Components, 1.20.5+）+ Codec / PacketCodec**，**不用旧 NBT tag**（1.21.1 大改点）。
- 控制器方块实体存"节点图"：节点列表 + 边列表 + 电网缓冲。
- config：GSON + **configVersion**，缺键取默认值，改结构就升版本（照 yongye）。

## 七、Phase 规划

- **Phase 0** 工程骨架：Loom 工程、注册框架（方块 / 方块实体 / 物品 / ScreenHandler / 配方序列化器）、config(configVersion)、mixin 占位、四件套文档骨架(README / HANDOVER / SKILL / 游玩介绍)。
- **Phase 1** 最小闭环：控制器方块 + 一个生产面板 + 内部缓冲 + 机架式 GUI，跑通「合成面板→装进控制器→出货」。
- **Phase 2** 电力：发电面板、耗电、电网缓冲、电不足暂停、电量显示。
- **Phase 3** 加速模块 + 升级槽 + 多档面板（基础 / 进阶 / 终极）。
- **Phase 4** 面板品类扩充（矿 / 农 / 刷怪 / 加工链）+ 数据组件持久化 + 与相邻容器对接导出。
- **Phase 5** 面板互联：把 MVP 的"目标选择"升级为节点-边数据模型下的正式连线关系。
- **Phase 6+** 节点画布渲染：自绘节点卡片 + 贝塞尔连线 + 拖拽 / 缩放；控制 / 逻辑面板（过滤 / 分流 / 条件）。

## 九、未决项（待作者拍板）

1. 电单位命名（焦耳 J / 电 / RF-like）。
2. 面板是"物品插进控制器"还是"直接在画布里 new 出来的虚拟节点"？（影响能否被偷、能否热插拔）。
3. 采集类面板要不要"锚定真实区块 / 生物群系条件"（更硬核）还是纯抽象产出（更省事）。
4. 数值基调：偏休闲（产量高、发电简单）还是偏硬核（必须搭发电链）。
5. 是否跨维度 / 是否需要强制加载区块。
